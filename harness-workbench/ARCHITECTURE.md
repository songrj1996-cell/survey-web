# 系统全景图：这个平台是怎么运转的

> 面向所有团队成员（含非技术人员）的系统解读。不涉及具体代码，只讲清楚"谁负责什么、数据怎么流、整体怎么跑"。

---

## 一、这是什么系统？

这是一个**调研分析工作台**。

用户把问卷数据（Excel / CSV）或评论数据上传进来，平台会自动帮你读懂数据、做统计、生成一份完整的分析报告，还能让你用自然语言追问。最后报告可以导出为 Word、PDF，或者直接发进飞书文档。

---

## 二、系统支持三种工作模式

| 模式 | 是什么 | 几步走 |
|------|--------|--------|
| **问卷分析**（定性/标准模式） | 上传一张问卷回收表，AI 读懂每列是什么题，生成分析报告 | 5 步 |
| **定量分析**（跑数表模式） | 同时上传问卷、清数表、跑数表，AI 直接读统计数据写报告 | 4 步 |
| **评论分析** | 上传一张评论或用户反馈表，AI 做主题提炼、情感分类，生成简报 | 3 步 |

另外还有一个独立的**数据标注**功能：检测问卷回收数据中有没有疑似 AI 作答或低质量作答，并对数据质量打分。

---

## 三、系统整体结构

用一句话概括：用户在网页上操作 → 浏览器调用后端接口 → 后端驱动 AI（Dify）做分析 → 结果实时推送给浏览器展示。

```
┌──────────────────────────────────────────────────────────────┐
│                        用户的浏览器                           │
│   上传文件 / 点击按钮 / 实时看到报告生成过程                   │
└────────────────────────┬─────────────────────────────────────┘
                         │  HTTP / 实时推送（SSE）
                         ▼
┌──────────────────────────────────────────────────────────────┐
│                    后端服务（FastAPI）                         │
│                                                              │
│  ┌─────────────┐   ┌─────────────┐   ┌──────────────────┐  │
│  │   接口层     │   │   业务层     │   │    存储层         │  │
│  │  routers/   │──▶│  services/  │──▶│  data/ 文件夹     │  │
│  │ 接收请求     │   │ 编排分析流程  │   │ 文件/JSON 持久化  │  │
│  └─────────────┘   └──────┬──────┘   └──────────────────┘  │
│                           │                                  │
└───────────────────────────┼──────────────────────────────────┘
                            │  调用 AI 接口
                            ▼
┌──────────────────────────────────────────────────────────────┐
│                       Dify AI 服务                            │
│       （外部 AI 应用平台，提供多个专用 AI 应用）                │
│                                                              │
│  题型识别 / 方案规划 / 报告写作 / 主题提炼 / 评论分类 / ...    │
└──────────────────────────────────────────────────────────────┘
```

## 四、核心模块分别负责什么

### 1. 接口层（routers/）— 前台服务员

相当于餐厅前台，负责接收每个请求、验证用户有没有权限、把任务传给后厨，再把结果端回给用户。

- `survey.py` — 受理问卷分析的每一步操作（上传、识别、方案、统计、报告、追问）
- `crosstab.py` — 受理定量/跑数表模式的文件上传
- `comment_analysis.py` — 受理评论分析（上传、预处理、运行）
- `annotate.py` — 受理数据标注（上传、确认、检测、打标、下载）
- `export.py` — 受理导出报告（Word / PDF / Markdown / 飞书）
- `history.py` — 受理查看历史报告（列表、详情、标题修改）
- `admin.py` — 受理管理员操作（管理用户、查看审计日志）
- `settings_api.py` — 受理设置管理（提示词、界面文案、系统开关）
- `feishu.py` — 负责飞书登录的整个过程（授权、回调、登出）

### 2. 业务层（services/）— 后厨

真正"做事情"的地方。每个服务模块负责一项业务的全流程编排。

- `survey_service.py` — 问卷分析五步流程的总指挥
- `crosstab_service.py` — 跑数表模式：解析问卷、清数表、跑数表，建立列定义
- `report_engine.py` — 报告生成引擎：组装提示词，多轮调用 AI，把每一章拼成完整报告
- `report_render.py` — 报告渲染车间：把 AI 写出的 Markdown 文本转成 Word / PDF / 飞书格式
- `report_history.py` — 报告归档：生成完成后自动写入历史记录
- `comment_upload.py` — 评论上传：接收文件，创建 session
- `comment_preprocess.py` — 评论预处理：列识别、清洗、分层抽样（上限 15000 条）
- `comment_run.py` — 评论分析运行：驱动完整分析流水线
- `comment_pipeline.py` — 评论分析流水线：筛相关→提主题→合并去重→情感分类→摘原文
- `annotate_workflow.py` — 数据标注流程：AI 找疑似 AI 作答→人工确认→质量打标
- `history_service.py` — 历史记录查询：列表筛选、详情读取
- `settings_service.py` — 设置管理：提示词 / 文案 / 系统开关的读写
- `auth.py` — 认证辅助：登录态验证、权限判断
- `feishu_auth.py` — 飞书 OAuth state 管理：生成和验证授权跳转参数

### 3. 存储层（storage/ + data/ 文件夹）— 档案室

项目**没有数据库**，业务状态主要存在本地 JSON 文件里；上传的原始文件和导出结果以文件形式存在 data/ 的子目录中。

| 路径 | 存什么 |
|------|--------|
| `data/sessions/<uuid>.json` | 每次分析的工作区（上传的数据、中间结果、最终报告），2 小时后自动清理 |
| `data/history.json` | 所有已完成报告的归档（标题、时间、报告正文、编号 R-001, R-002…） |
| `data/whitelist.json` | 有权限使用系统的用户列表（谁能用哪个功能） |
| `data/prompts.json` | 发给 AI 的提示词模板（管理员可修改） |
| `data/audit_logs.json` | 操作记录（谁、什么时候、做了什么） |
| `data/app_settings.json` | 系统开关配置（如重复文件提醒是否开启） |
| `data/ui_texts.json` | 页面文案（管理员可自定义修改） |
| `data/web_logins.json` | 飞书登录态缓存 |
| `data/comment_uploads/` | 评论分析上传的原始文件（临时） |
| `data/annotate_results/` | 数据标注完成后导出的 Excel 结果文件 |

### 4. 核心工具库（core/）— 基础设施

所有业务都要用的公共能力，不包含具体业务逻辑：

- `config.py` — 统一读取系统配置（AI 密钥、文件路径、各种阈值）
- `security.py` — 判断当前用户有没有权限做某件事
- `parsing.py` — 把上传的 Excel / CSV 文件解析成系统可处理的二维表
- `responses.py` — 标准化响应格式（包括实时推送格式 SSE）

### 5. 外部集成（integrations/）— 对外接口

与外部服务通信的专用封装，只做协议转换，不写业务逻辑：

- `dify_client.py` — 调用 Dify AI 平台，支持实时流式输出（AI 边想边输出，用户能看到打字效果）
- `feishu_client.py` — 调用飞书开放平台，实现登录验证 + 把报告推送到飞书文档

### 6. 前端（static/）— 门店界面

负责页面展示、流程步骤管理和用户交互；真正的数据处理、权限判断、AI 调用和报告归档全部由后端完成。

- `index.html` — 唯一的主页面，所有三个分析模式都在这里
- `login.html` — 飞书登录页
- `js/core/core.js` — 底层工具：发请求、接收 AI 实时流输出、工具函数
- `js/features/` — 每个功能的交互逻辑（问卷分析、评论分析、标注、报告展示、历史记录、设置）

## 五、数据是怎么流转的（以问卷分析为例）

```
用户上传 Excel 文件
        │
        ▼
[后端] 解析 Excel → 存入 session 文件（临时工作区）
                              │
                              ▼
[AI] 题型识别：每一列是什么角色？（年龄/性别/评分/开放题…）
    → 结果存回 session
                              │
                              ▼
[AI] 方案规划：报告要写哪几章？每章聚焦什么问题？
    → 方案存回 session
                              │
                              ▼
[后端] 统计计算：频数、占比、交叉分析 → 纯数字结果存回 session
                              │
                              ▼
[AI] 多轮分章写报告：把统计数字 + 原始数据 → 喂给 AI
    → AI 分章逐一输出（实时流式推送给浏览器）
    → 完整报告存回 session
                              │
                              ▼
     报告生成完成 → 系统自动归档到 history.json（每用户最多保留最近 20 条，超出自动裁剪旧记录）
                              │
                    ┌─────────┴─────────┐
                    ▼                   ▼
              用户追问             导出 Word/PDF/飞书
         （调 AI 继续对话，        （重新渲染报告格式）
          历史记录同步更新）
```

**关键特点：**
- 浏览器看到的"AI 一边打字一边生成"效果，是通过 **SSE（服务端推送）** 实现的——后端每产出一段文字就立刻推给浏览器，不等全部写完再返回。
- session 是临时工作区，超过 2 小时可能在系统清理时被删除（不是精确定时，而是下次启动或创建新 session 时触发清理）。history 是系统正式归档区，但每用户最多保留最近 20 条，超出会自动裁剪旧记录。报告生成完成后会自动进入历史，不需要用户手动保存。

## 六、session 和 history 有什么区别？

这是理解系统数据流的关键：

| | session（临时工作区） | history（永久归档） |
|---|---|---|
| **存在时间** | 超过 2 小时可能被系统清理（触发时机：下次启动或新建 session） | 每用户最多保留最近 20 条，超出裁剪旧记录 |
| **存什么** | 上传的原始数据、中间分析结果、当前报告草稿 | 报告正文、标题、创建时间、编号 |
| **什么时候创建** | 用户上传文件时 | 报告生成完成时，系统自动写入 |
| **作用** | 支撑当前这次分析的整个过程 | 供后续查阅、追问、导出 |
| **评论分析** | 同样使用 session 临时存储抽样数据和分析结果 | 评论报告生成后也自动进入历史 |

简单说：**session 是草稿桌，history 是档案柜**。报告写完会自动归档，草稿桌会定期清理。

## 七、谁能用这个系统？权限是怎么设计的

1. **登录方式**：通过飞书企业账号授权登录；是否强制登录可由配置控制，开发/测试环境可关闭。
2. **权限控制**：管理员维护一份白名单，每个人可以被单独授权哪些功能（问卷分析 / 评论分析 / 数据标注）。没在白名单里的飞书用户登录后什么功能都用不了。
3. **管理后台**：白名单管理员可以在设置页面增删用户、调整权限、修改 AI 提示词、修改界面文案、控制系统开关、查看操作审计日志。

---

## 八、管理员后台能做什么？

系统除了分析功能外，还有一套完整的管理后台，管理员登录后可以：

| 功能 | 具体能做什么 |
|------|------------|
| **用户管理** | 添加/删除白名单用户，给每个人分配能用哪些功能模块 |
| **提示词管理** | 修改发给 AI 的提示词模板，调整 AI 的写作风格或分析逻辑，不需要改代码 |
| **界面文案管理** | 修改页面上显示的说明文字、引导语等，不需要改代码 |
| **系统开关** | 控制某些行为是否启用，例如重复上传同一文件时是否弹出提醒 |
| **审计日志** | 查看所有用户的操作记录（谁、什么时候、用了什么功能） |

## 九、AI 是怎么接进来的？

本系统**不直接调用 OpenAI / Claude 等大模型**，而是通过 **Dify**（一个 AI 应用编排平台）来使用 AI 能力。Dify 上配置了多个专用 AI 应用，按使用场景分组如下：

**问卷分析（标准模式）**

| AI 应用 | 负责什么 |
|---------|----------|
| 题型识别应用 | 看每列数据，判断是什么类型的题目 |
| 方案规划应用 | 根据题目设计，规划报告的章节结构 |
| 标准报告写手 | 接收统计数据，分章节撰写报告正文 |

**定量分析（跑数表模式）**

| AI 应用 | 负责什么 |
|---------|----------|
| 跑数表方案规划 | 读问卷原文，生成语义化的章节大纲 |
| 大样本报告写手 | 处理跑数表数字，撰写大样本定量报告 |

**大样本开放题分析**

| AI 应用 | 负责什么 |
|---------|----------|
| 主题提炼应用 | 从大量开放题回答中提取主要话题 |
| 主题合并应用 | 把相似的话题合并、去重 |
| 评论分类应用 | 判断每条回答的情感倾向和所属话题 |

**评论舆情分析**

| AI 应用 | 负责什么 |
|---------|----------|
| 评论分析应用 | 一个 Workflow 完成：相关性筛选→主题提取→情感分类→简报生成 |

**数据标注**

| AI 应用 | 负责什么 |
|---------|----------|
| AI 作答检测应用 | 找出问卷中疑似 AI 作答或低质量作答的条目 |
| 质量打标应用 | 给数据打质量分 |

系统通过环境变量（`.env` 文件）管理每个 Dify 应用的访问密钥，管理员在服务器上配置一次，代码里不写死任何密钥。

## 十、技术选型的白话解释

| 选型 | 是什么 | 为什么这么选 |
|------|--------|-------------|
| **FastAPI**（后端框架） | Python 写的 Web 框架 | 原生支持实时推送（SSE），适合 AI 流式输出场景 |
| **JSON + 文件存储** | 数据存在文本文件和本地目录里，不用数据库 | 部署简单，不需要安装维护数据库；数据量小，够用 |
| **Dify** | AI 应用编排平台 | 可以图形化配置 AI 流程、管理 prompt、随时调整，不需要改代码 |
| **原生 JavaScript**（前端） | 不用 React/Vue 等框架 | 轻量，不需要构建打包，部署直接放静态文件 |
| **SSE 推送** | 服务器实时向浏览器推文字 | 让用户能"看着 AI 写报告"，体验流畅 |
| **飞书登录** | 通过飞书企业账号认证 | 对企业内部使用场景方便，无需维护独立账号体系 |

## 十一、系统的边界和局限

- **无数据库**：历史报告、用户列表都存在本地 JSON 文件，适合单机/小团队使用。大规模并发或需要复杂查询时会遇到瓶颈。
- **session 非精确定时删除**：session 超过 2 小时后，会在系统下次启动或创建新 session 时被清理，不是持续运行的定时器。分析过程中长时间不操作，工作区可能消失，需要重新上传分析。
- **history 有上限**：每用户最多保留最近 20 条历史记录，超出后旧记录会被自动裁剪。
- **依赖 Dify 和飞书**：AI 能力和登录都依赖外部服务，这两个服务如果不可用，核心功能无法使用。
- **单机部署**：目前设计为部署在一台内网服务器上，具体地址以部署配置为准。
