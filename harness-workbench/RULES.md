# 代码改动规则（AI 行为约束）

本文件是所有参与此项目的协作者（包括 AI 助手和人类开发者）的行为准则。所有规则在任何对话、任何任务中无条件生效。

---

## 一、改动前必须确认

**在动任何代码文件之前，必须先向用户说明以下三点，得到明确同意后才能开始：**

1. 打算改哪个文件（精确到文件名）
2. 具体改什么（改的内容是什么）
3. 为什么要改（用户要求了什么）

没有用户的明确确认，不得修改任何代码文件，无论改动多小。

---

## 二、发现问题只报告，不擅自修改

在阅读或修改代码过程中，如果发现了用户没有提到的问题（bug、不一致、潜在风险、可优化之处），**只能告知用户，不能自行修改**。

正确做法：
> "我注意到 X 文件里有一处可能的问题：[描述]。需要我帮你修复吗？"

错误做法：顺手把它改掉，或者在完成主任务的同时附带修复它。

---

## 三、改动范围严格等于用户要求

用户说改 A，就只改 A。

不允许以下行为，即使理由听起来合理：
- "顺便把相关的 B 也改了"
- "A 和 C 是一起的，我把 C 也更新了"
- "这里原来的写法不太好，我顺手重构了一下"

**未经要求的任何改动，一律不做。**

---

## 四、一个目标可以有多个必要改动，但必须先说清楚

如果用户的一个目标需要同时修改多个文件，可以一次性完成这些必要改动，但必须在动手前先说明：

1. 涉及哪些文件
2. 每个文件分别改什么
3. 为什么这些文件都属于当前目标的必要范围
4. 改完后准备怎么验证

用户确认后，才能按这个范围执行。

执行过程中如果发现新的问题、新的优化点或额外风险，只能先报告，不能顺手纳入本次改动。

---

## 五、架构分层不能打破

本项目采用分层架构，层与层之间不能随意跳层调用：

- **routers/**：只负责接收 HTTP 请求、校验参数、调用 service、返回响应。不能在这里写业务逻辑或直接读写文件。
- **services/**：负责业务流程编排。调用 storage、integrations、core 工具库，不能直接操作 HTTP 响应格式。
- **storage/**：只负责读写 `data/` 目录下的本地数据文件，不负责业务流程判断。
- **core/**：放所有业务都可能用到的通用能力，例如配置、权限、解析、响应格式。不能反向依赖 routers 或 services。
- **integrations/**：只封装外部服务调用，例如 Dify、飞书。不能写具体业务流程。
- **schemas/**：只放请求、响应、数据结构定义，不写业务逻辑。
- **static/js/core/**：放前端通用能力，例如请求、SSE、工具函数。
- **static/js/features/**：放具体功能页面的交互逻辑。

新功能必须遵守这个分层，不能图省事把逻辑堆在 router 里，也不能让底层模块反向依赖上层模块。

后端分层相关改动完成后，应优先运行：

```bash
python scripts/check_boundaries.py
```

---

## 六、data/ 目录下的真实数据禁止删除、清空或覆盖

`data/` 目录存放的是运行时真实数据（历史报告、用户白名单、审计日志、登录状态、提示词、用户上传文件、标注结果等）。

**任何情况下不得删除、清空、覆盖或格式化这些文件和目录**，包括但不限于：

- `data/history.json`
- `data/whitelist.json`
- `data/audit_logs.json`
- `data/prompts.json`
- `data/app_settings.json`
- `data/ui_texts.json`
- `data/web_logins.json`
- `data/sessions/`
- `data/comment_uploads/`
- `data/annotate_results/`

如果某个任务确实需要操作这些文件，必须先单独说明：

1. 要操作哪个文件或目录
2. 会做什么操作
3. 是否会影响现有数据
4. 是否已经有备份或可恢复方案

经用户明确确认后才能进行。

---

## 七、启动服务或运行测试前，要注意是否会写入 data/

本项目的某些入口在启动或执行时可能会读写 `data/`，例如启动 FastAPI 应用时可能清理过期 session 文件。

因此，运行以下操作前，如果可能触发 `data/` 变更，必须先提醒用户：

- 启动后端服务
- 导入 `app.main`
- 跑会触发应用启动逻辑的测试
- 执行清理 session、写日志、写历史记录、写登录状态的脚本或接口

不能为了验证方便，擅自清理或改写运行时数据。

---

## 八、密钥和配置只走 .env + config.py

所有 API Key、外部服务地址、环境相关路径、业务阈值配置，只能通过 `.env` 文件 + `app/core/config.py` 统一读取。

不允许在任何代码文件中硬编码密钥、外部服务 URL、部署路径或敏感配置项。

内部前后端路由地址可以写在代码里，但应保持集中、清晰，不能散落成难以维护的魔法字符串。

---

## 九、不引入新的第三方依赖

不得在代码中 import 任何当前 `requirements.txt` 中不存在的第三方包。

如果确实需要新依赖，必须先向用户说明：

- 需要什么包
- 为什么需要
- 会带来什么影响

得到确认后，同步更新 `requirements.txt`，再写使用该包的代码。

---

## 十、修改部署敏感文件前必须单独提醒

以下文件的修改会直接影响服务器部署、启动、鉴权、外部服务连接或运行环境。修改前必须额外说明：

> 此修改会影响部署或运行环境。

部署敏感文件包括：

- `requirements.txt`
- `.env`
- `.env.example`
- `deploy.sh`
- `nginx-survey.conf`
- `server.py`
- `app/main.py`
- `app/core/config.py`
- `app/core/security.py`
- `app/integrations/dify_client.py`
- `app/integrations/feishu_client.py`

---

## 十一、改完必须说明验证方式

每次代码改动完成后，必须告诉用户已经做了哪些验证，以及结果是什么。

常见验证包括：

- 后端分层检查：`python scripts/check_boundaries.py`
- Python 语法检查：`python -m compileall app`
- 前端页面改动：浏览器或页面冒烟检查
- 接口改动：说明验证了哪个接口、输入是什么、结果是什么

如果没有运行验证，必须明确说明原因，不能假装已经验证。

---

## 十二、Git 操作必须由用户明确要求

不得主动执行以下 Git 操作，除非用户明确要求：

- `git commit`
- `git push`
- `git reset`
- `git checkout -- <file>`
- 删除、回滚或覆盖未确认的用户改动

查看状态、查看差异可以用于理解当前工作区，但不能擅自提交、重置或清理。

---

## 十三、密钥、隐私和日志不能泄露

不得在回复中直接展示以下内容：

- `.env` 原文
- API Key
- Cookie
- 登录 token
- 飞书授权信息
- 用户上传数据中的敏感原文

如果必须引用日志或配置内容，敏感字段必须打码，只保留足够定位问题的信息。
