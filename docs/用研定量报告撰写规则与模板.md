# 用研定量报告撰写规则与模板

适用对象：游戏内问卷、测服问卷、正式服问卷、英雄/系统/功能体验类定量调研报告。

参考文件：
- `【用研一部】MLBB 测服手感优化调研小结 202604.pdf`
- `卡蒂塔翻新前置调研20250701.md`

## 一、整体写作原则

### 1. 先给业务判断，再给数据过程

报告开头必须先回答业务方最关心的问题：

- 这个功能/角色/系统整体表现好不好？
- 哪些维度是优势，哪些维度是短板？
- 问题是体验价值不足，还是认知/使用引导不足？
- 后续优化应优先解决什么？

不要一开始堆题目结果。题目结果应服务于核心判断。

### 2. 结论必须同时包含“评价水平 + 问题方向 + 证据”

推荐句式：

> 玩家对【对象】的【维度】评价【较高/中等/较低】，满意度为【X.XX】，主要问题集中在【问题1】（XX%）和【问题2】（XX%）。结合主观反馈，问题本质是【体验机制/表达方式/认知引导/操作成本】。

不推荐：

> 玩家满意度为 3.55，问题是 A、B、C。

原因：后者只有数据罗列，没有解释问题性质。

### 3. 所有核心结论都要能回溯到数据

每条核心结论至少包含以下证据中的两类：

- 均值/占比/频数。
- 人群差异。
- 主观反馈归因。
- 典型原话。

如果只有主观原话，没有量化占比，应标注为“少数玩家”“个别玩家”“主观反馈中出现”。

### 4. 先总体，再拆人群，再解释原因

每个模块固定采用：

1. 总体表现。
2. 分人群差异。
3. 主要问题排序。
4. 主观反馈归因。
5. 可执行启示。

### 5. 主观题不是附录素材，而是解释定量结果的证据

主观题应承担两类作用：

- 解释为什么某个指标高/低。
- 把抽象选项翻译成具体体验问题。

写法上应先归纳观点，再放原话：

> 主观反馈中，玩家对“模型不够精致”的反馈主要集中在三类：模型过时、动作僵硬、色彩单调。代表性原话如下：

不要直接堆大量原话。

## 二、推荐报告结构

### 版本 A：完整定量报告

适合英雄翻新、系统重做、较大功能评估。

```markdown
# 【项目名】调研报告 / 调研小结

**调研目的**：说明本次调研要支持什么业务决策。

**样本情况**：有效样本量、国家/区服/平台/关键人群构成。

**分析方法**：问卷来源、投放场景、定量/定性结合方式。

**问卷投放时间**：X月X日-X月X日。

**调研人员**：姓名。

**数据说明**：关键人群定义、评分口径、样本量不足提示。

## 核心结论

### 维度一：【例如美术表现】

- 一句话判断整体表现。
  - 关键指标 1：均值/占比，主要问题。
  - 关键指标 2：均值/占比，主要问题。
  - 结合主观反馈说明问题本质。

### 维度二：【例如玩法评价】

- 一句话判断整体表现。
  - 玩法机制/平衡性/操作体验等关键指标。
  - 主要问题、问题占比、人群差异。
  - 结合主观反馈说明优化方向。

### 维度三：【例如文本/世界观/认知】

- 一句话判断整体表现。
  - 满意度/认知率/认同度。
  - 最希望保留和最期待优化的内容。
  - 需要注意的认知错位或设定风险。

## 整体评价

- 总结各维度的相对强弱。
- 明确优先级：优先修什么、保留什么、观察什么。

## 模块 1：【维度名称】

### 子模块 1：【具体对象】

- 总体而言，玩家对【对象】的【指标】为【X.XX/XX%】。
- 分人群来看，【人群 A】的评价相对更高/更低。
- 总体而言，玩家认为主要问题是【问题1】（XX%），其次是【问题2】（XX%）。
- 主观反馈中，玩家对【问题1】的反馈主要为：
  - 归因 1：说明。
  - 归因 2：说明。
  - 归因 3：说明。

> 代表性原话 1（中文翻译）
> 代表性原话 2（中文翻译）

### 子模块 2：【具体对象】

同上。

## 模块 2：【维度名称】

同上。

## 样本人口学/游戏属性情况

- 国家/段位/常玩位置/使用频率/是否体验过相关功能。
- 如样本量不足，明确说明不做定量拆分。

## 附录：数据表及主观题原文

- 数据表链接。
- 主观题原文链接。
```

### 版本 B：短版调研小结

适合测服功能验证、上线前快速判断。

```markdown
# 【项目名】调研小结

**调研目的**：调研玩家对【功能/版本内容】的评价。

**样本情况**：总有效样本【N】份。

**分析方法**：游戏内【测服/正式服】投放问卷，进行定量分析。

**问卷投放时间**：X月X日-X月X日。

**调研人员**：姓名。

## 核心结论

📌 【一句话总判断】：已使用/已体验玩家对【功能】评价【较高/中等/较低】，但【认知/开启/触达/使用】仍存在缺口，说明【功能本身有价值，但需要加强引导/入口/默认设置/规则解释】。

- 认知/开启/体验漏斗：XX% 注意到，XX% 开启，XX% 实际使用。
- 满意度/认同度：实际使用玩家满意度为 X.XX，设计目的认同度为 X.XX。
- 主要阻碍：未开启/未使用首要原因是【原因1】（XX%），其次是【原因2】（XX%）。
- 主要体验问题：主观反馈集中在【问题1】、【问题2】、【问题3】。

## 1. 使用/开启情况和未开启原因

- 总体而言，XX% 的玩家开启/使用了【功能】。
- 分人群来看，【人群 A】未开启比例较高。
- 未开启原因较为集中/分散，首要原因是【原因】（XX%）。
- 业务解释：这说明阻碍主要来自【认知不足/操作成本高/习惯迁移困难/功能价值不明确】。

## 2. 功能认知和体验评价

### 2.1 更新及功能认知

- XX% 玩家明确注意并了解具体内容，XX% 完全没有注意到。
- 需要注意的是，【手动开启/隐藏入口/低频场景】功能的认知和使用比例低于默认开启功能。

### 2.2 满意度评价

- 清楚了解并实际使用过功能的玩家，满意度均值为 X.XX。
- 分人群来看，【人群 A】满意度相对更高/更低。

### 2.3 设计目的认同度评价

- 玩家对设计目的认同度为 X.XX。
- 如果满意度高但使用率低，应判断为“功能价值成立，但触达/引导不足”。
- 如果使用率高但满意度低，应判断为“功能被需要，但体验实现不足”。

## 3. 主观反馈

- 玩家主要反馈的体验不佳问题是：
  - 问题 1：一句话概括。
  - 问题 2：一句话概括。
  - 问题 3：一句话概括。

> 代表性原话 1（中文翻译）
> 代表性原话 2（中文翻译）

## 样本人口学/游戏属性情况

## 附录：数据表及主观题原文
```

## 三、模块写法规则

### 1. 核心结论写法

每条核心结论建议控制在 1 个主判断 + 2-4 个证据点。

推荐结构：

```markdown
**【维度名称】**

- 玩家对【对象】的【维度】评价【判断】，核心原因是【原因概括】。
  - 【指标1】为【X.XX/XX%】，主要问题是【问题1】（XX%）和【问题2】（XX%）。
  - 分人群来看，【人群 A】评价更高/更低，说明【解释】。
  - 结合主观反馈，玩家实际不满集中在【具体体验】。
```

### 2. 人群差异写法

只写有解释价值的人群差异，不机械罗列所有交叉结果。

推荐句式：

- 分段位来看，高段位玩家对【问题】更敏感，选择【选项】的比例显著较高，说明该问题更影响高强度对局体验。
- 分常用分路来看，【分路】玩家未开启比例较高，可能与其对【操作/视野/反应】的要求有关。
- 分使用频率来看，经常使用【对象】的玩家评价更高，说明熟悉度会影响接受度。

### 3. 主观反馈归因写法

主观反馈不要只按问卷选项复述，要二次归因。

推荐格式：

```markdown
主观反馈中，玩家对【问题】的反馈主要集中在：

- 【归因 1】：具体说明。
- 【归因 2】：具体说明。
- 【归因 3】：具体说明。

代表性原话：
> 原话（翻译）
```

### 4. 风险提示写法

当少数玩家情绪强烈、涉及流失风险、付费/高投入玩家受损、核心机制被否定时，需要单列风险提示。

推荐句式：

> **风险提示**：有少数玩家对【机制】表达强烈负面情绪，核心不满在于【原因】。虽然样本占比不高，但该反馈来自【高投入/高熟悉度/关键人群】玩家，建议关注其对【留存/口碑/功能接受度】的影响。

## 四、常用判断框架

### 1. 满意度与使用率交叉判断

| 使用/认知 | 满意度 | 解释 | 报告建议 |
| --- | --- | --- | --- |
| 高 | 高 | 功能价值和触达均成立 | 建议保留方向，做细节优化 |
| 低 | 高 | 功能本身好，但触达不足 | 强调引导、入口、默认开启、说明 |
| 高 | 低 | 玩家需要该功能，但实现不佳 | 优先修体验问题 |
| 低 | 低 | 需求和方案均需验证 | 谨慎推进，补充定性或 AB 测试 |

### 2. 问题优先级判断

优先级按以下顺序判断：

1. 影响核心体验且占比高。
2. 占比不高但来自高价值/高投入玩家。
3. 影响认知、开启、使用漏斗。
4. 只是偏好差异或低频建议。

### 3. 结论措辞强度

- 70% 以上：可写“绝大多数”“压倒性认可”。
- 50%-70%：可写“多数”“整体较正面”。
- 30%-50%：可写“相当比例”“较多玩家”。
- 10%-30%：可写“部分玩家”。
- 10% 以下：可写“少数玩家”“个别玩家”。

如果样本量很小，优先写频数，避免只写百分比。

## 五、输出风格规范

### 1. 语言风格

- 用“玩家认为/玩家反馈/玩家对……评价”作为主语。
- 避免过度主观判断，例如“这个设计很好”“这个问题很严重”。
- 用“说明/表明/可能意味着”连接数据和业务解释。
- 结论要直接，但不要超过数据能支持的范围。

### 2. 数据呈现

- 均值保留两位小数。
- 百分比保留两位小数，样本很小时可同时写频数。
- 满意度、认同度、难度等评分必须说明满分口径，或在数据说明中统一定义。
- 分人群结论只写样本量足够、差异有解释价值的结果。

### 3. 引用原话

- 原话要服务于观点，每个观点放 2-5 条即可。
- 外语原话后附中文翻译。
- 原话前必须有归纳标题，不要无结构堆叠。

## 六、报告质量检查清单

提交前逐项检查：

- 是否开头交代了调研目的、样本、方法、时间、人员。
- 核心结论是否能在 1 分钟内让业务方知道“好不好、问题在哪、下一步做什么”。
- 每个模块是否都有总体数据、人群差异、主要问题和主观归因。
- 是否区分了“功能价值问题”和“认知/使用引导问题”。
- 是否对少数强烈负面反馈做了风险提示。
- 是否避免机械罗列题目结果。
- 是否把主观题加工成观点，而不是原话堆叠。
- 是否说明了样本量不足或不适合拆分的地方。
- 附录是否提供数据表和主观题原文链接。
