# Survey Web 前端设计规范

这是一个调研分析工作台，不是营销网站。所有新增前端功能都应该服务于“上传数据、确认信息、生成报告、查看结果、继续追问”这类高频工作流。

设计目标是：克制、清晰、专业、轻量、有反馈。新功能应该像原本就属于当前产品。

## 整体气质

- 做成数据分析工具，而不是宣传页。
- 页面要紧凑但不拥挤，信息层级清楚，方便反复操作。
- 视觉重点放在当前任务、下一步操作、处理状态和结果内容。
- 使用深浅主题都成立的 token 化设计，不做只适配单一主题的颜色。
- 不做大 hero、不做营销卡片堆叠、不做大面积装饰图形。

## 颜色

使用浅色工作台为基础，同时支持深色主题。

基础颜色规则：

- 主背景：深色模式接近深蓝黑，浅色模式为浅灰蓝。
- 主容器：比背景略亮，保持低对比、低噪音。
- 次级容器：再亮一层，用于上传区、卡片、配置区。
- 主强调色：紫色到蓝紫色渐变。
- 辅助强调色：青色用于信息或 AI 感知，绿色用于成功，红色用于错误，琥珀色用于警告。
- 边框：低透明度细边框，不要用重边框。
- 文字：主文字高对比，说明文字降一级，辅助说明再降一级。

实现时优先使用这些 CSS 变量：

- 背景与容器：`--bg`、`--surface`、`--surface-2`、`--glass`、`--glass-hover`
- 边框：`--border`、`--border-act`
- 强调色：`--accent`、`--accent-2`、`--accent-soft`、`--accent-glow`
- 文本：`--text`、`--text-2`、`--text-3`
- 状态色：`--green`、`--red`、`--amber`、`--cyan`

不要新增脱离这套变量的主色。临时状态色也应从这些变量派生。

## 圆角、阴影和动效

- 常规控件圆角使用 8px 左右。
- 普通卡片和内容块圆角可用 12-14px。
- 上传区、较大的容器可以稍大，但不要超过当前产品的柔和圆角感。
- 阴影只用于 hover、抽屉、弹层、上传区聚焦等状态，不要让页面到处漂浮。
- 动效控制在 0.2s 左右，主要用于 hover、展开、进入、状态变化。
- hover 可以轻微抬升或发光，但不要夸张。

## 页面骨架

工作台页面使用固定结构：

- 左侧是导航栏。
- 右侧是主工作区。
- 主工作区按流程切换 panel。
- 每个 panel 顶部是步骤条，下面是标题和说明。
- 操作按钮通常放在内容块底部右侧。

新增页面时，默认采用：

```html
<section class="panel">
  <div class="step-bar">...</div>
  <div class="panel__header">
    <h1 class="panel__title">页面标题</h1>
    <p class="panel__desc">一句说明当前步骤要做什么</p>
  </div>
  <!-- 页面内容 -->
</section>
```

标题不要太大。当前产品内页标题是工作台标题，不是宣传标题。

## 边距与留白

边缘间距是界面整洁的基础，必须统一，不能让元素贴边。

规则：

- 主工作区左右 padding：32px（`.main { padding: 28px 32px }`）。
- 撑满宽度的 toolbar / bar 类组件（如 report-toolbar、step-bar），左右 padding 必须与主工作区对齐，保持 **32px**。如果 panel 已通过负 margin 撑满，toolbar 自身的 padding 同样要写 32px。
- 内嵌卡片或弹层内部水平 padding：16–20px。
- 任何按钮、控件不应紧贴容器边缘，至少保留 24px 的视觉距离。
- Toolbar 内的小操作按钮（导出、图标按钮等）缩小一档：`padding: 5px 12px; font-size: 12px`，避免在高度固定的 bar 里显得过大。

## 步骤条

所有流程都使用横向步骤条。

设计规则：

- 当前步骤使用紫色文字和底部边线。
- 已完成步骤可点击回看，颜色比当前步骤弱。
- 未到达步骤禁用或不可点击。
- 步骤编号使用小圆点。
- 窄屏允许换行，不要挤压重叠。

文案规则：

- 每个步骤名用 2-6 个中文字。
- 动词或状态要明确，例如 `上传数据`、`数据确认`、`方案确认`、`生成报告`、`报告 & 追问`。

## 按钮

按钮是当前界面的主要行动入口，必须统一。

样式规则：

- 主按钮使用紫色到蓝紫色渐变，白字。
- 次按钮使用半透明玻璃底、细边框、浅文字。
- 危险操作只在 hover 或明确状态中使用红色。
- 禁用状态降低透明度，取消 hover 抬升和阴影。
- 按钮高度保持轻量，不要做厚重的大按钮，除非是最终下载/完成动作。

使用规则：

- 每个操作区最多一个主按钮。
- 次要操作放在主按钮左侧。
- 主按钮文案必须具体，例如 `开始分析`、`确认方案`、`下载结果`。
- 避免只写 `提交`、`确定`、`下一步`，除非上下文非常明确。
- 按钮里有图标时，图标尺寸约 14px，和文字保持 6-8px 间距。

## 上传区

上传是这个产品的关键入口，必须使用统一的上传区视觉。

### 单文件上传

单文件上传使用大面积虚线上传区。

视觉规则：

- 容器为虚线边框。
- 背景使用半透明玻璃感。
- 图标居中，尺寸约 52px。
- 主文案 15px 左右，说明文案 12px 左右。
- hover 或拖拽进入时，边框变为强调色，背景变为强调色浅底，并出现柔和光晕。
- 上传中不改变整体结构，只把图标替换成 spinner，文案改为正在上传。

推荐文案：

- 主文案：`拖放文件到这里，或点击选择`
- 说明文案：`支持 CSV / Excel（最大 50MB）`
- 上传中：`正在上传 文件名...`

### 多文件上传

多文件上传也必须使用同一套上传语言，不能露出原生文件 input 作为主要 UI。

推荐结构：

- 外层是一个上传组，纵向排列。
- 每个文件位是一个“上传卡片/上传行”，样式接近小型 upload-zone。
- 每个上传卡片包含图标、文件角色、说明、已选择文件名。
- 未选择时是虚线边框和提示文案。
- 已选择时使用强调色边框或浅色强调底，显示文件名和可重新选择入口。
- 所有必填文件选完后，底部主按钮才启用。

倍市得/跑数表三文件上传必须按这个规则设计：

- `问卷文件`
- `回答数据（清数结果）`
- `跑数表（统计结果）`

不要使用浏览器默认文件按钮作为用户看到的主要控件。

## 卡片和选择项

选择类入口使用扁平卡片。

视觉规则：

- 卡片背景使用主容器色。
- 边框 1-1.5px。
- 左侧可放图标方块，图标容器约 40px。
- 标题 15px 左右，说明 13px 左右。
- hover 时边框变强调色，背景出现强调色浅底，允许轻微外发光。
- 禁用项降低透明度，不响应 hover。

用于：

- 分析类型选择。
- 任务选择。
- 多文件上传里的文件位。
- 设置项选择。

不要把普通页面区块都做成卡片。卡片只用于“可选择的一项”或“重复列表中的一项”。

## 表单和输入

表单应轻量、紧凑、可扫读。

规则：

- 输入框背景使用 `--input-bg` 或接近 `--surface` 的颜色。
- 边框使用 `--border`。
- focus 时边框变强调色。
- placeholder 使用弱文字色。
- label 用 12-13px，清楚说明字段含义。
- 表单行之间保持 8-12px 间距。
- 长文本输入高度要足够，但不要占满首屏。

不要在一个页面里混用多套 input 风格。

## 表格和列表

数据预览、历史记录、确认列表都要方便扫读。

表格规则：

- 表头使用强调色浅底或主容器色。
- 行分隔用低透明边框。
- hover 行轻微变亮。
- 长文本单元格要省略或限制宽度，避免撑破布局。
- 数字、时间、ID 使用稳定宽度或等宽字体。

列表规则：

- 一行一项时使用 `.col-row` 风格：浅背景、细边框、小圆角。
- 编号或状态标签放左侧。
- 主信息在中间，操作在右侧。
- 内容长时允许换行，但不能挤压操作按钮。

## 抽屉和弹层

设置、历史、管理类内容使用右侧抽屉。

规则：

- 抽屉从右侧进入。
- 背后有半透明遮罩。
- 顶部有标题和关闭按钮。
- 内容区滚动，头部固定。
- 抽屉内可以使用左侧二级导航加右侧内容。

不要用新页面打断主流程，除非用户正在进入完全不同的模块。

## Toast 和反馈

所有用户可感知结果都要有反馈。

规则：

- 成功：绿色。
- 错误：红色。
- 普通提示：强调色。
- 长任务开始、完成、失败都要提示。
- 表单校验失败要直接说明原因。

不要只在 console 输出错误。

## 加载和长任务

AI 分析、报告生成、批量标注、上传解析都属于长任务。

规则：

- 使用 spinner 表示正在处理。
- 使用进度块或状态文字说明当前阶段。
- SSE 流式输出时，应实时刷新可见内容。
- 长任务页面不能空白等待。
- 如果失败，保留用户已输入内容，并给出可重试路径。

推荐状态文案：

- `正在上传...`
- `AI 正在规划分析方案，请稍候`
- `报告生成中，请耐心等待`
- `最后更新：HH:mm:ss，当前步骤：...`

## 报告和追问区

报告页是结果工作区，应优先保证阅读和继续追问。

规则：

- 报告正文居中占主要宽度。
- 目录可以放左侧。
- 追问面板放右侧，可折叠，可加宽。
- 报告 Markdown 使用统一排版，不要临时写内联样式。
- 导出操作放在标题区域右侧或报告顶部。

不要让追问输入框挤压报告正文到不可读。

## 中文文案

整体使用简洁中文。

规则：

- 标签具体，不要抽象。
- 能写任务名就不要写泛化动词。
- 描述文字一到两句即可。
- 错误信息要告诉用户哪里错了。
- 上传类文案要说明文件类型、大小限制、文件角色。

推荐：

- `上传问卷文件`
- `上传回答数据`
- `上传跑数表`
- `拖放文件到这里，或点击选择`
- `支持 CSV / Excel（最大 50MB）`
- `请把问卷、回答数据、跑数表三个文件都选上`

避免：

- `提交`
- `确定`
- `点击这里`
- `文件 1`
- `文件 2`
- `操作失败`

## 响应式

必须保证桌面和窄屏都可用。

规则：

- 900px 以下主内容减少 padding。
- 640px 以下侧边栏可隐藏。
- 横向步骤条允许换行。
- 表格和宽内容允许横向滚动。
- 右侧追问栏在窄屏变为下方区域。
- 按钮文字不能溢出容器。

不要用随视口线性缩放的字体大小。

## 实现约定

实现时优先使用这些现有类名和概念：

- `.layout`
- `.sidebar`
- `.main`
- `.panel`
- `.panel--hidden`
- `.panel__header`
- `.panel__title`
- `.panel__desc`
- `.step-bar`
- `.step-bar__item`
- `.btn`
- `.btn--primary`
- `.btn--ghost`
- `.upload-zone`
- `.analysis-type-card`
- `.drawer`
- `.toast`
- `.spinner`

交互优先使用这些函数或同类模式：

- `$()`
- `showToast()`
- `esc()`
- `renderMarkdown()`
- `goStep()`
- `setViewStep()`
- `renderStepBars()`
- `consumeSSE()`
- `consumeSSEPost()`

只有当这些规则无法判断细节，或需要避免破坏已有行为时，才去读取现有代码确认。

## 检查清单

完成前检查：

1. 新 UI 是否像当前产品的一部分。
2. 是否复用了现有按钮、上传区、步骤条、面板、抽屉或表格语言。
3. 上传 UI 是否没有裸露原生文件 input。
4. 深色和浅色主题是否都可读。
5. hover、loading、disabled、error、success 状态是否完整。
6. 中文文案是否具体、简洁、无乱码。
7. 窄屏是否不重叠、不溢出。
8. 长任务是否有可见进度或反馈。

## 明确禁止

- 禁止把原生文件 input 当作主要上传 UI。
- 禁止新增一套和当前产品无关的颜色系统。
- 禁止为单个按钮写一次性样式。
- 禁止做营销式 hero 或大面积装饰背景。
- 禁止把页面区块层层套卡片。
- 禁止静默失败。
- 禁止在中文乱码状态下编辑并保存文件。
