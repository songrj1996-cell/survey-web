# 调研分析业务流程规范

这个 skill 负责“业务怎么走”，不负责“界面长什么样”。如果任务主要是颜色、按钮、上传区、布局和视觉一致性，使用前端设计 skill；如果任务涉及分析流程、步骤流转、数据处理、AI 分工、报告生成或导出历史，就使用这个 skill。

## 核心原则

保持产品的业务边界稳定：

- Python 负责确定性数据处理、解析、统计、校验、文件读写和导出。
- LLM/Dify 负责语义理解、方案规划、主观题聚类、报告写作和追问回答。
- 统计数字不能让 LLM 自己编、猜、重算。
- 用户必须能在关键业务节点确认结果，例如题型、方案、AI 识别结果。
- 长流程必须能持久化、可回看、可导出、可追问。

## 什么时候使用

使用本 skill 处理这些任务：

- 新增一种分析模式。
- 修改定性分析、跑数表分析、数据标注的步骤。
- 改上传文件要求、必填文件、上传后进入哪一步。
- 改 AI 在何时介入、输入什么、输出什么。
- 改 Python 统计、跑数表解析、主观题原文收集。
- 改报告生成结构、报告免责声明、报告导出。
- 改追问逻辑、历史报告续问、Dify conversation 选择。
- 改 session/history 持久化字段。
- 改飞书导出、Word/PDF/Markdown 导出。
- 改权限、白名单、审计日志对业务流程的影响。

不要用本 skill 处理纯视觉问题，例如按钮不好看、上传区样式不一致、响应式错位。这些属于前端设计规范。

## 当前三类业务流程

### 定性分析

定性分析是 5 步：

1. 上传数据。
2. 数据确认，也就是确认题型、题名、选项、量表范围、矩阵题结构等。
3. 方案确认，也就是 AI 规划分析章节和分析问题，用户确认或修订。
4. 生成报告，也就是 Python 计算统计和整理开放题，LLM 写报告。
5. 报告 & 追问，也就是展示报告、导出、飞书文档、历史保存和继续问答。

定性分析适用于普通问卷原始数据。平台可以基于原始数据做题型识别、统计计算和开放题分析。

### 倍市得/跑数表分析

跑数表分析是 4 步：

1. 上传三个文件：问卷文件、回答数据（清数结果）、跑数表（统计结果）。
2. 方案确认：AI 读问卷原文和可用题目，生成章节大纲和报告结构问题。
3. 生成报告：Python 直接读取跑数表数字，收集开放题原文，LLM 做主观题聚类和报告写作。
4. 报告 & 追问：展示报告、导出、飞书文档、历史保存和继续问答。

跑数表分析必须跳过“数据确认”。上传成功后自动建列，直接进入方案确认。

跑数表分析的硬边界：

- 跑数表中的统计数字是事实来源。
- 平台不重新计算跑数表已有统计。
- LLM 不生成、猜测、修正统计数字。
- Python 只解析和渲染跑数表，必要时校验结构。
- 主观题可以由平台收集原文并交给 LLM 聚类。

### 数据标注

数据标注是独立流程：

1. 上传数据。
2. 列确认 & 任务选择。
3. AI 作答识别。
4. AI 识别结果确认。
5. 回答质量打标。
6. 完成 & 下载。

数据标注可以使用 AI 帮助识别和打标，但关键判断需要用户确认，尤其是高概率 AI 作答名单。

## Python 与 LLM 分工

### Python 负责

Python 做可验证、可重复的工作：

- 文件上传、类型检查、大小限制。
- CSV / Excel 解析。
- 跑数表解析。
- 问卷文本提取。
- 自动建列。
- 题型结构校验。
- 统计计算或跑数表数字渲染。
- 开放题原文收集。
- session/history 读写。
- 导出 Word / PDF / Markdown / 飞书文档。
- 权限、登录、白名单、审计日志。

### LLM/Dify 负责

LLM 做语义和写作工作：

- 题型识别辅助。
- 方案规划。
- 方案修订。
- 主观题主题提取、合并、分类、摘要。
- 报告写作。
- 报告追问。
- 数据标注中的语义判断。

### 禁止

- 禁止让 LLM 输出最终统计数字作为事实来源。
- 禁止让 LLM 替代跑数表解析。
- 禁止在跑数表模式中用平台自算结果覆盖跑数表数字。
- 禁止缺少用户确认就把高风险 AI 判断直接写入最终结果。

## 上传与建列规则

### 定性分析上传

定性上传只需要一个问卷数据文件。

上传后：

1. 后端解析文件。
2. 生成 session。
3. 前端进入数据确认。
4. 后端触发题型识别。
5. 用户确认列配置后才能进入方案规划。

### 跑数表分析上传

跑数表上传必须包含三个文件：

- 问卷文件。
- 回答数据（清数结果）。
- 跑数表（统计结果）。

上传后：

1. 后端解析问卷文件，得到问卷原文和题目映射。
2. 后端解析回答数据，用于主观题原文和必要画像信息。
3. 后端解析跑数表，用于最终统计数字。
4. 后端自动建列：开放题为 `open_text`，画像维度为 `profile_dim`，其他不需要平台统计的列可设为 `ignore`。
5. session 标记为 `mode = crosstab`。
6. 直接进入方案确认，不进入数据确认。

如果三个文件缺失任意一个，不能开始分析。

## 方案确认规则

方案确认是用户和 AI 对报告结构达成一致的节点。

### 定性方案

定性方案可以包含：

- 分析章节。
- 每章节包含的题目或列。
- 交叉分析设计。
- 开放题分析问题。
- 待用户确认的问题。

定性方案可依赖用户确认后的题型和列结构。

### 跑数表方案

跑数表方案只做报告结构规划，不做数据口径确认。

跑数表 planner 输入：

- 问卷原文。
- 实际有统计结果的题目清单。
- 开放题清单。
- 用户修订意见（修订时）。

跑数表 planner 输出：

- `parts`: 章节数组，每个章节包含 `name` 和 `scope`。
- `open_questions`: 0-3 个报告结构层面的待确认问题。

跑数表 planner 禁止提出数据类问题，例如：

- 题型是否正确。
- 样本口径是否正确。
- 是否要重新计算。
- 某列是否应参与统计。
- 某个数字是否可信。

这些都不属于 planner 的职责。

## 报告生成规则

### 定性报告

定性报告生成时：

1. Python 根据确认后的列配置计算统计。
2. Python 收集开放题文本。
3. LLM 根据统计结果、开放题分析和方案写报告。
4. 报告完成后保存到 session 和历史记录。

### 跑数表报告

跑数表报告生成时：

1. Python 将跑数表解析结果渲染为 `stats_md`。
2. Python 不重新计算跑数表已有统计数字。
3. Python 收集开放题原文。
4. LLM 对开放题做主题聚类。
5. Writer 使用跑数表数字、问卷原文、章节方案和主观题聚类结果写报告。
6. 报告完成后保存 `mode` 和 `analyst_app`，供追问和历史导出选择正确 Dify 应用。

跑数表报告结构默认：

1. 核心结论。
2. Part 1 受访者画像。
3. 方案中规划的其他章节。
4. 行动建议。

如果存在满意度、好评率或类似核心指标，必须优先进入核心结论靠前位置。

## 主观题聚类规则

主观题聚类用于从开放题原文中抽取主题和代表性反馈。

推荐流程：

1. 分批提取主题。
2. 合并去重主题。
3. 回跑分类并标注情感。
4. 汇总每个主题的计数、占比、正负摘要和代表原文。

代表原文规则：

- 优先保留真实原文。
- 可以附中文翻译。
- 不足时展示已有原文，不编造。
- 每个主题可按正向、负向、中性各取若干条。

## 追问规则

报告追问必须绑定正确的报告上下文。

当前 session 追问：

- 使用当前报告内容。
- 使用当前报告对应的 analyst app。
- 保留 conversation id，避免每次追问丢上下文。

历史报告追问：

- 从 history 读取报告内容、mode、analyst_app、conversation id。
- 根据历史记录的 `analyst_app` 或 `mode` 选择正确 Dify key。
- 不要把定性报告的 conversation 用到跑数表应用，反之亦然。

如果 conversation 不可用或跨应用失效，应能重新建立对话，而不是让用户卡死。

## 历史记录与 session

长流程必须保存关键业务状态。

session 应保存：

- session id。
- 上传文件解析结果。
- mode。
- columns / confirmed columns。
- plan。
- stats_md。
- report_md。
- analyst_app。
- conversation id。
- owner 信息。

history 应保存：

- 历史 id。
- 报告标题和编号。
- 报告 Markdown。
- mode。
- analyst_app。
- owner 信息。
- QA 记录。
- 导出相关信息。

session 可以有 TTL，但 history 是用户可回看的长期记录。

多 worker 或服务重启场景下，不要只依赖内存 session。

## 导出规则

支持导出：

- Word。
- PDF。
- Markdown。
- 飞书文档。

导出必须使用与当前报告模式匹配的报告预处理逻辑。

定性报告：

- 保留定性分析所需的完整免责声明。

跑数表报告：

- 使用跑数表模式的简化免责声明。
- 不要混入旧定性免责声明。

历史报告导出：

- 根据 history 中的 mode 或 plan.mode 判断导出规则。
- 不要只根据当前 session 判断。

飞书文档导出：

- 使用登录用户或机器人身份的实现要与当前权限配置匹配。
- 导出完成后应返回可访问链接。
- 失败时暴露清晰错误信息，便于定位权限、token 或接口问题。

## 权限与审计

业务功能要尊重登录和权限边界。

规则：

- 未登录用户不能访问受保护页面和 API。
- 非白名单用户不能进入业务流程。
- 管理功能只对 admin 可见。
- 用户只能看到自己有权限的历史记录。
- 关键操作写审计日志，例如登录、上传、生成报告、导出、管理用户。

权限失败时：

- 页面请求可以跳转登录。
- API 请求应返回 JSON 错误。
- 前端应给用户可理解提示。

## 新增分析模式时

新增分析模式前先回答：

1. 它需要几个上传文件？
2. 是否需要数据确认？
3. 哪些数字由 Python 计算，哪些数字来自外部文件？
4. 哪些任务交给 LLM？
5. 是否需要方案确认？
6. 是否需要主观题聚类？
7. 报告结构与现有模式是否一致？
8. 追问应该使用哪个 Dify 应用？
9. history 需要保存哪些新字段？
10. 导出和免责声明是否需要分流？

默认不要复制整条旧流程。只保留该模式真正需要的步骤。

## 修改流程时的检查清单

完成任何业务流程改动前检查：

1. 定性 5 步是否仍然可走通。
2. 跑数表 4 步是否仍然跳过数据确认。
3. 数据标注流程是否没有被误伤。
4. Python 和 LLM 分工是否清晰。
5. 统计数字来源是否可靠。
6. session 是否保存了后续步骤需要的字段。
7. history 是否保存了导出和追问需要的字段。
8. 当前报告和历史报告是否都能导出。
9. 当前报告和历史报告是否都能追问。
10. 错误提示是否能让用户知道下一步怎么处理。

## 明确禁止

- 禁止让跑数表模式重新走数据确认。
- 禁止在跑数表模式中重算已有统计数字。
- 禁止让 LLM 编造统计数字或代表原文。
- 禁止新增分析模式时只改前端入口，不补齐 session、history、导出和追问。
- 禁止只支持当前 session，不支持历史报告导出或续问。
- 禁止关键 AI 判断绕过用户确认。
- 禁止权限失败时只返回模糊错误。
